1. 概述

任务脚本是一个 Node.js 程序，由 Airdrop Farm 以子进程方式启动，自动化执行区块链交互操作（如领水、签到、跨链桥等）。

一个脚本从开发到运行的完整路径：

📝 编写脚本 → 📦 打包 zip → 📤 上传市场 → ⬇️ 用户安装 → ⚙️ 填写配置 → ▶️ 启动任务 → 📋 查看日志

2.2 任务创建与执行流程

  1. 用户选择脚本：从已安装列表中选择一个任务模板脚本。
  2. 表单渲染：前端读取脚本的 manifest.json → schema 字段，通过 jsonSchemaToFieldMeta() 将其转换为表单字段 FieldMeta[]。嵌套对象会被展平为 parent.child 点号键名。
  3. 账户选择：如果 manifest.json 中声明了 requiredAccountTemplateIds，前端会加载匹配模板的账户列表供用户勾选，支持按账号池筛选和批量模式。
  4. 提交：前端将表单值通过 unflattenDotNotation() 还原为嵌套对象后，调用 injectAccountData() 将选中的账户数据注入 _accounts 字段，然后调用 taskApi.create()。
  5. 持久化：task:create handler 将配置写入 SQLite tasks 表，状态为 idle。
  6. 启动：用户点击播放按钮 → task:start → TaskService.startTask()。
  7. 执行：TaskService 将 task.config 注入为环境变量（TASK_CONFIG / TASK_{KEY} 等），以 child_process.spawn() 运行脚本入口文件。
  8. 输出：stdout → info 日志，stderr → error 日志，实时推送到渲染进程并存入 task_logs 表。
  9. 结束：exit code 0 → complete，非 0 → error。

3. 脚本项目结构

my-airdrop-script/
├── manifest.json      必需 脚本元数据和配置 Schema
├── index.js           必需 入口文件（须与 manifest.entryPoint 一致）
├── package.json       可选 npm 依赖声明
└── utils.js           可选 其他辅助文件
⚠️ 约束：manifest.json 和入口文件必须位于 zip 包根目录。打包时不要在 zip 中嵌套一层文件夹。
⚠️ 不支持原生 C++ 模块：由于 Electron 内嵌的 Node.js ABI 与独立 Node.js 不一致，better-sqlite3、canvas、sharp、bcrypt 等含原生编译的包将直接崩溃。请只用纯 JS 包，或使用 esbuild / webpack 打包。

4. manifest.json — 脚本清单文件

4.1 字段速查表

字段类型必需说明
idstring是 全局唯一标识，推荐反向域名风格，如 com.example.my-script
namestring是 人类可读的显示名称
versionstring是 语义化版本号，如 "1.0.0"
descriptionstring是 脚本功能简述
entryPointstring是 入口文件名，相对于脚本目录，如 "index.js"
runtimestring是 必须是 "node"。暂无其他运行时
schemaobject是 JSON Schema 格式的配置表单定义。必须包含 "type": "object" 和 "properties": {...}
requiredAccountTemplateIdsstring[]可选 需要的账号模板 ID 列表。声明后脚本才能获取账户数据。
permissionsstring[]可选 声明的权限："network"、"filesystem"。仅供信息展示。
tagsstring[]可选 市场分类标签
changelogstring可选 版本更新日志

4.2 Schema → 表单的转换规则

JSON Schema 字段生成的表单控件
type: "string" + enum下拉选择框
type: "string"（无 enum）文本输入框
type: "number" / "integer"数字输入框（支持 minimum / maximum）
type: "boolean"复选框
type: "object" 含 properties子属性被展平为 parent.child 点号键名
title字段标签
default默认值
description帮助文本
顶层 required 数组必填校验
pattern正则校验

5. 配置注入 — 脚本如何获取参数

脚本通过环境变量接收配置：

环境变量说明
TASK_CONFIG 完整配置 JSON 字符串。保留了类型（数字就是数字，布尔就是布尔）。这是推荐方式。
TASK_{KEY} 每个配置键的大写形式。值始终是字符串。如 TASK_RPCURL、TASK_CHAINID。
嵌套键展开为单层：TASK_PROXY.HOST 不存在，只有 proxy.host 键展开为 TASK_PROXY_HOST。
TASK_ID 当前任务的 UUID
NODE_PATH 指向脚本目录下的 node_modules
⚠️ 重要：始终使用 TASK_CONFIG 而非单独的 TASK_* 变量！后者将所有值强制转为字符串（"3" 而非 3，"true" 而非 true）。

读取配置的标准写法

// ✅ 推荐：使用 TASK_CONFIG，类型完整
const config = JSON.parse(process.env.TASK_CONFIG || '{}')

const rpcUrl = config.rpcUrl          // string
const chainId = config.chainId        // number
const headless = config.headless      // boolean
const waitBetween = config.waitBetweenTx  // number
const proxyHost = config.proxy?.host  // string（嵌套对象还原了）

console.log(`连接到 ${rpcUrl}，链 ID: ${chainId}`)

// ❌ 避免：单独的环境变量丢失类型
const count = process.env.TASK_WAITBETWEENTX  // "5" — 字符串！
const num = parseInt(count, 10)               // 必须手动解析

6. 访问账户数据

如果你的脚本需要操作用户的账户（钱包地址、私钥等），需要通过以下机制获取：

6.1 前置条件

  1. 在 manifest.json 中声明 requiredAccountTemplateIds，指定需要的账号模板 ID。
  2. 用户必须在「账户管理」页面中基于对应模板创建了账号，并将它们分配到某个账号池。
  3. 用户在创建任务时勾选要使用的账号。

6.2 单任务模式 — 所有账号在一个任务中

用户勾选多个账号后，前端 injectAccountData() 将它们注入到 config._accounts 数组中：

const config = JSON.parse(process.env.TASK_CONFIG || '{}')
const accounts = config._accounts || []

for (const account of accounts) {
  console.log(`账号 ID: ${account.id}`)
  console.log(`账号池: ${account.pool}`)
  console.log(`标签: ${account.labels.join(', ')}`)

  // account.data 包含模板定义的字段（地址、私钥等）
  const { address, privateKey } = account.data
  console.log(`地址: ${address}`)

  // 使用 ethers.js 签名交易
  const { Wallet, JsonRpcProvider } = require('ethers')
  const provider = new JsonRpcProvider(config.rpcUrl)
  const wallet = new Wallet(privateKey, provider)
  // ...
}

console.log(`共处理 ${accounts.length} 个账号`)
process.exit(0)

6.3 批量模式 — 每个账号一个独立任务

用户在创建任务时开启批量模式，前端为每个账号创建一个独立任务，将单个账号数据注入 _account_id、_account_data、_account_pool：

const config = JSON.parse(process.env.TASK_CONFIG || '{}')

const accountId = config._account_id        // string
const accountData = config._account_data    // object（私钥等）
const accountPool = config._account_pool    // string

console.log(`正在为账号 ${accountId} (池: ${accountPool}) 执行任务`)
const { address, privateKey } = accountData
// ...
process.exit(0)

字段类型说明

templateIdstring所属模板 ID
dataobject模板定义的字段（如 address、privateKey、mnemonic）
poolstring账号池名称
labelsstring[]标签列表
notesstring备注

7. 关于代理 ⚠️ 重要限制

⚠️ 当前限制：系统不会自动将用户配置的代理列表注入到任务脚本的配置中。脚本无法通过 _proxies 或类似机制获取用户在「代理管理」中保存的代理。

目前脚本获取代理的两种方式：

方式一：通过 Schema 让用户手动填写代理参数（推荐）

在你的 manifest.json 的 schema 中定义一个嵌套的 proxy 对象字段，让用户在前端表单中手动输入代理参数：

{
  "schema": {
    "type": "object",
    "properties": {
      "proxy": {
        "type": "object",
        "title": "代理配置",
        "properties": {
          "host": { "type": "string", "title": "代理主机" },
          "port": { "type": "number", "title": "代理端口", "minimum": 1, "maximum": 65535 },
          "username": { "type": "string", "title": "用户名" },
          "password": { "type": "string", "title": "密码" }
        }
      }
    }
  }
}

脚本中读取：

const config = JSON.parse(process.env.TASK_CONFIG || '{}')
const proxy = config.proxy
if (proxy?.host) {
  const proxyUrl = `http://${proxy.username}:${proxy.password}@${proxy.host}:${proxy.port}`
  // 配置 puppeteer / playwright / axios 等使用此代理
}

方式二：通过 package.json 依赖数据库直连（高级）

脚本如果打包了 better-sqlite3（注意：需要静态编译版本，不能用标准 npm 包），可以直接读取 Airdrop Farm 的 SQLite 数据库查询代理表：

// ⚠️ 高级用法，需要处理跨平台和 Electron ABI 问题
const Database = require('better-sqlite3')
const { app } = require('electron')  // 脚本运行在 Electron 环境中
const { join } = require('path')

const db = new Database(join(app.getPath('userData'), 'airdrop-farm.db'))
const proxies = db.prepare('SELECT * FROM proxies WHERE status = ?').all('active')
console.log(`找到 ${proxies.length} 个活跃代理`)
⚠️ 风险提示：方式二存在严重的跨平台兼容性问题，因为 better-sqlite3 包含原生 C++ 模块，在 Electron 环境中通常无法正常加载。不推荐此方式。

8. 日志与输出

8.1 基本规则

  • stdout → info 级别日志。每个 console.log() 产生一条。
  • stderr → error 级别日志。每个 console.error() 产生一条。
  • 行缓冲：输出按行拆分，每行一条日志。
  • 日志实时显示在任务详情面板中。
  • 日志持久化到 SQLite task_logs 表，可通过 API 查询。
  • 前端保留最近 500 条日志在内存中滚动显示。
  • stdout/stderr 完整内容在任务结束后保留最后 10,000 字符，可通过 task:getOutput 获取。

8.2 退出码

Exit Code任务状态
0complete — 成功完成
任何非零值error — 异常终止
被信号杀死error — 进程被终止

8.3 进度报告（无专用 API）

当前系统没有结构化的进度 API（TaskProgress 接口虽定义了 percent 和 message 字段，但未曾从脚本侧填充）。用日志消息模拟进度：

const total = 100
for (let i = 0; i < total; i++) {
  await doWork()
  if (i % 10 === 0) {
    console.log(`[进度] ${Math.round(((i + 1) / total) * 100)}%`)
  }
}
console.log('[进度] 100% — 完成')
process.exit(0)

9. 生命周期与进程控制

9.1 状态转换

操作实际行为Linux/macOSWindows
启动 以spawn() 子进程方式运行入口文件。自动安装 npm 依赖。 idle → running
暂停 冻结进程 / 暂停输出流 SIGSTOP（真冻结） 软暂停（输出流暂停，进程继续运行）
恢复 恢复进程 / 恢复输出流 SIGCONT 恢复流
停止 SIGTERM → 5 秒后 SIGKILL running / paused → stopped

9.2 优雅退出

let shuttingDown = false

process.on('SIGTERM', () => {
  if (shuttingDown) return
  shuttingDown = true
  console.log('收到停止信号，正在优雅退出...')
  // 关闭数据库连接、清理临时文件等
  process.exit(0)
})

// 如果有未捕获异常，确保输出有意义的信息然后退出
process.on('uncaughtException', (err) => {
  console.error(`未捕获异常: ${err.message}`)
  process.exit(1)
})
⚠️ 暂停行为不可靠：Windows 上 SIGSTOP 不可用，系统仅暂停输出流的捕获，你的脚本仍然在内部继续执行。不要依赖暂停来精确控制执行时机。将脚本设计为可被 SIGTERM 终止即可。

10. 依赖管理

10.1 npm 安装时机

如果脚本目录包含 package.json，TaskService.startTask() 在首次启动时自动执行 npm install --omit=dev --no-audit --no-fund（180 秒超时）。

后续启动跳过安装，除非：

  • node_modules 不存在
  • package.json 比 node_modules 更新（修改过）
  • 某个声明的依赖在 node_modules 中缺失（不完整检测）

10.2 推荐的可用的包

纯 JavaScript 包均可安全使用：

用途推荐包
EVM 交互ethers、viem、web3
Solana 交互@solana/web3.js、@solana/spl-token
SUI 交互@mysten/sui
HTTP 请求axios、node-fetch（Node 18+ 内置 fetch）
加密 / 助记词bip39、@noble/hashes、ethereum-cryptography
CLI 美化chalk、ora（注意 stdout 捕获影响）
定时 / 重试p-retry、p-limit
💡 最佳实践：使用 esbuild 或 webpack 将脚本和所有依赖打包为单个 .js 文件。这样无需 package.json，彻底规避 npm install 和原生模块问题。

10.3 esbuild 打包示例

# 安装 esbuild
npm install -D esbuild

# 打包为单文件
npx esbuild src/index.js --bundle --platform=node --outfile=dist/index.js --external:electron

# 打包后的 dist/index.js 可以直接作为入口点

manifest.json

{
  "id": "com.example.ping-checker",
  "name": "Ping 检查器",
  "version": "1.0.0",
  "description": "定时检查 URL 的可达性并记录响应时间",
  "entryPoint": "index.js",

  "schema": {
    "type": "object",
    "properties": {
      "targetUrl": {
        "type": "string",
        "title": "目标 URL",
        "description": "要检测的网址"
      },
      "intervalSec": {
        "type": "number",
        "title": "检查间隔（秒）",
        "default": 60,
        "minimum": 10,
        "maximum": 3600
      },
      "maxChecks": {
        "type": "number",
        "title": "最大检查次数",
        "default": 10,
        "minimum": 1
      }
    },
    "required": ["targetUrl"]
  },
  "permissions": ["network"],
  "tags": ["utility", "monitoring"]
}

index.js

const config = JSON.parse(process.env.TASK_CONFIG || '{}')
const targetUrl = config.targetUrl
const interval = (config.intervalSec || 60) * 1000
const maxChecks = config.maxChecks || 10

let checkCount = 0
let shuttingDown = false

process.on('SIGTERM', () => {
  shuttingDown = true
  console.log('收到停止信号，完成当前检查后退出...')
})

async function check() {
  const start = Date.now()
  try {
    const res = await fetch(targetUrl, { signal: AbortSignal.timeout(10000) })
    const elapsed = Date.now() - start
    console.log(`[${new Date().toISOString()}] ✅ ${targetUrl} — HTTP ${res.status} — ${elapsed}ms`)
  } catch (err) {
    const elapsed = Date.now() - start
    console.error(`[${new Date().toISOString()}] ❌ ${targetUrl} — ${err.message} — ${elapsed}ms`)
  }
}

async function main() {
  console.log(`开始监控 ${targetUrl}，间隔 ${interval / 1000}s，最多 ${maxChecks} 次`)
  for (let i = 0; i < maxChecks && !shuttingDown; i++) {
    await check()
    checkCount++
    if (i < maxChecks - 1 && !shuttingDown) {
      await new Promise(r => setTimeout(r, interval))
    }
  }
  console.log(`监控结束。共执行 ${checkCount} 次检查。`)
  process.exit(0)
}

const config = JSON.parse(process.env.TASK_CONFIG || '{}')
const accounts = config._accounts || []

if (accounts.length === 0) {
  console.error('没有选择任何账户！请在创建任务时勾选要使用的 EVM 钱包账户。')
  process.exit(1)
}

async function claimFromFaucet(address, faucetUrl, amount) {
  console.log(`正在为 ${address} 申请 ${amount} ETH...`)
  const res = await fetch(faucetUrl, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ address, amount })
  })
  if (!res.ok) throw new Error(`HTTP ${res.status}: ${await res.text()}`)
  const data = await res.json()
  console.log(`  ✅ ${address} 领取成功 — txHash: ${data.txHash || 'N/A'}`)
  return data
}

async function main() {
  console.log(`=== EVM 批量领水开始 ===`)
  console.log(`钱包数量: ${accounts.length}`)
  console.log(`RPC: ${config.rpcUrl}`)
  console.log(`水龙头: ${config.faucetUrl}`)
  console.log(`数量: ${config.amount} ETH/钱包`)
  console.log('')

  let success = 0
  let failed = 0

  for (let i = 0; i < accounts.length; i++) {
    const account = accounts[i]
    const { address } = account.data
    console.log(`[${i + 1}/${accounts.length}] 处理 ${address} (池: ${account.pool})`)

    try {
      await claimFromFaucet(address, config.faucetUrl, config.amount)
      success++
    } catch (err) {
      console.error(`  ❌ 失败: ${err.message}`)
      failed++
    }

    if (i < accounts.length - 1) {
      await new Promise(r => setTimeout(r, config.delaySec * 1000))
    }
  }

  console.log('')
  console.log(`=== 完成 ===`)
  console.log(`成功: ${success}, 失败: ${failed}`)
  process.exit(failed > 0 ? 1 : 0)
}

14. 常见问题

Q: 为什么我的脚本获取不到账户数据？

检查以下几点：

  1. manifest.json 中是否声明了 requiredAccountTemplateIds？
  2. 用户是否在「账户管理」中基于对应模板创建了账号？
  3. 用户在创建任务时是否勾选了要使用的账号？
  4. 在脚本中通过 config._accounts 读取（而非 config.accounts —— 注意下划线前缀）。

Q: 为什么我的脚本获取不到代理？

这是已知限制。当前系统不会将用户在「代理管理」中保存的代理自动注入到脚本配置中。目前只能通过以下方式解决：

  1. 推荐：在 manifest.json 的 schema 中定义代理参数字段，让用户手动填写。
  2. 未来计划：系统可能会增加自动注入代理的功能。如有需求，请向开发团队反馈。

Q: 脚本能访问文件系统吗？

可以。脚本没有沙箱，拥有与 Electron 主进程相同的系统权限。它可以读写文件、发起网络请求、执行其他进程。

Q: 脚本之间能通信吗？

不能。每个任务在独立的子进程中运行，相互隔离。

Q: 能串联多个任务吗？

目前不支持工作流编排。没有内置的任务依赖或串联机制。可以通过定时任务（cron）进行时间上的协调。

Q: npm install 失败怎么办？

常见原因：

  • 依赖了原生 C++ 模块（如 better-sqlite3）→ 换成纯 JS 替代方案或使用 esbuild 打包
  • 网络问题 → 检查是否能访问 npm registry
  • 磁盘空间不足 → 清理后重试

Q: 如何调试脚本？

最佳方式是脱离 App 本地运行（见第 11 节）。设置 TASK_CONFIG 环境变量后直接 node index.js，使用你熟悉的调试工具。在 App 内运行时，只能通过日志输出排查问题。

Q: TypeScript 脚本可以用吗？

可以。入口文件以 .ts 结尾时，TaskService 会自动使用 npx tsx 运行。但需确保脚本目录的 node_modules 中有 tsx（通过 package.json 声明依赖）。建议打包为 .js 后再发布，减少运行时依赖。

Q: 定时任务 (Scheduled Task) 是如何工作的？

SchedulerService 每 10 秒 tick 一次，检查所有启用中的定时任务的 cron 表达式是否与当前时间匹配。匹配时自动创建新 Task 并启动。cron 表达式支持 5 字段标准格式（分 时 日 月 星期）。
